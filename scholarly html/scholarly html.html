1. Introduction

1.1 Purpose

The purpose of this website is to allow zoo staff to manage their animals as well potential users to view the zoo's animals and attraction

1.2 Document conventions

The pages were made using HTML (for the structure), PHP (for sessions) , JavaScript (for functionalities such as input validation and AJAX), CSS (for style) and Python (for web data gathering)

1.3 Intended Audience and Reading Suggestions

The website was designed mainly for zoo staff and visitors.

1.4 Product Scope

The site allow the normal user to search for animals by certain criteria (location, type, eating habit etc), see the atractions that the zoo give at their disposal, as well create or cancel appointments for those attractions. The admin user can add an animal or delete to the website, or not making it visible to the website.

2. Overall Description

2.1 Product Perspective

The final product is a zoo website, more precisely a web application (open-source) that works independently and is intended for general use. Being an open-source application, it is supposed to be used by an unlimited number of users, including a logging system, which brings a series of extra functionalities. The major components of the application are two: FRONTEND and BACKEND. The FRONT part (HTML, CSS, JavaScript) connects with the user, through it he can see and interact with the WEB server, thus benefiting from all the functionalities of the application. The BACK part (Xampp,Php,Python) ensures the correctness of the functionalities, being the actual implementation area. Of course, a MySQL database is used as a way to store the data included in the product component

2.2 Product Functions

        This application offers the user a series of functionalities:

  • Register
  • Autentification
  • Deconnect
  • Change Password
  • Add Animals
  • Remove animals
  • Account validation
  • Create / Cancel appointments
  • Searching for an animal by filters
  • User Session
        It should be mentioned that all these functionalities are only available for users registered and logged into the application.

2.3 User Classes and Characteristics

        The Zoo Connect application is intended for all web application users. Due to the functionalities it has, it is especially addressed to users who want to find out information about different categories of animals. It is considered that the user has the basic knowledge of internet operation and access to a device connected to the network.

2.4 Operating Environment

        Being a web application, Zoo Connect can be accessed from all operating systems (Windows, Linux and MacOS) respectively from any type of browser. These operating environments consist of a user interface generally provided by an application manager, respectively an application programming interface (API) provided by the application manager. The application works on any device/PC with standard specifications, even minimal ones, depending on the speed of the processor, RAM, ROM. The device must be connected to the Internet

2.5 Design and Implementation Constraints

        Regarding the design elements, a UI/UX Design will be used that will facilitate easy use by all people interested in the site. It is the duty of the developers to comply with some established specifications by common agreement regarding the color palette, font size, image sizes, spacing.
The chosen color palette :        Also, there are constraints on the Backend side: each team member must write code in the initially established languages.
Another technical constraint for the good collaboration of the developer team is the regular update (push) on git of any type of information, be they implementations and not only. Working in parallel on the same product is possible be quite demanding and it is recommended to pay more attention to the merging of individually worked code parts.

2.6 User Documentation

        The application falls into the category of web applications, and its use will be intuitive, more precisely suggested by the entire interface that facilitates the user's navigation in the application and the discovery of useful commands.

2.7 Assumptions and Dependencies

        If the user loses the connection to the Internet, then inevitably the web application will stop responding and its functionalities will become unavailable.
        SOFTWARE Dependency: If the application developers do not pay the necessary attention to the written code, especially when they use APIs, it threatens the performance and security of the application.
        The connection to the database must be secure, its structuring is very important because it is the basis for registration and authentication, changing the password.

3.External Interface Requirements

3.1 User Interfaces

      The interfaces between the website and a user are present through a set of architectural elements present in the web page. User login and authentication system represents a means by which their information such as identification data, search history on the website, favorite albums are kept confidential.

3.2 Hardware Interfaces

       From a hardware point of view, the website must be accessed from a device that corresponds to the norms of normal operation, being also connected to the Internet.

3.3 Software Interfaces

       From a software point of view, the website is connected to a MySQL database, and can be accessed from all operating systems (Windows, Linux) from high-performance browsers such as Firefox or Google Chrome. Like messages with other software components, when the user searches for information, the website obtains information from the database through queries, information that will be displayed on the page in the user's browser following the communication between the server and its browser. . The website will be available only to users who access the www service offered by the Internet based on the TCP/IP protocol.       

4. System Features

4.1 Register


Real time checking

The user clicks on the boxes where the data must be entered - The system allows the user to enter text in the respective boxes.
The user presses the login button, but has not filled in all the required fields - The system does not allow registration and warns the user that he has omitted a field that must be filled in.
The user presses the login button, but enters a non-existent username in the database - The system does not allow logging in and warns the user that he has entered a non-existent username.
The user presses the login button, but enters a wrong password – The system does not allow logging in and warns the user that he has entered a wrong password.
The user presses the login button, and all fields are filled in correctly – The system allows the user to successfully log in to the site.


Functional Requirments

Taking the text from the username and password input fields
Request from frontend to backend.

4.2 Autentification


Real time checking

The user clicks on the boxes where the data must be entered - The system allows the user to enter text in the respective boxes.
The user must complete the fields with his first name, last name, password, email and phone number. The system does not allow registration and warns the user that he has omitted a field that must be filled in.
The user presses the register button, but has not filled in all the required fields - The system does not allow registration and warns the user that he has omitted a field that must be filled in.
The user presses the register button, but enters an existent username in the database - The system does not allow logging in and warns the user that he has entered a non-existent username.
The user presses the register button, but enters a wrong password in the "Confirm Password" field – The system does not allow logging in and warns the user that he has entered a wrong password.
The user presses the register button, and all fields are filled in correctly – The system allows the user to successfully register to the site.


Functional Requirments

Taking the text from the name, password, email and phone number input fields
Request from frontend to backend.

4.3 Deconnect


The component allows a logged-in user to log out of the site.

The user presses the "Log out" button - The system logs the user off the site.

Request from frontend to backend.

4.4 Change Password


Real time checking

The user clicks on the boxes where the data must be entered - The system allows the user to enter text in the respective boxes.
The user must complete the fields with his last password and his new password. The system does not allow the change and warns the user that he has omitted a field that must be filled in.
The user presses the change button, but has not filled in all the required fields - The system does not allow change and warns the user that he has omitted a field that must be filled in.
The user presses the change button, but enters the same password - The system does not allow changing in and warns the user that he has entered a non-existent username.


Functional Requirments

Taking the text from the password and new password input fields
Request from frontend to backend.

4.5 Add Animals


Upon selecting an animal from the filters page, a post request is sent to the
animal page, which will search for the JSON corresponding to the animal selected by the user.
The animal page will open up, populate the specific section of the page with the animal's information found
in the JSON file (name, habitat, description etc).

4.6 Remove Animals


The page checks if the current user is an admin(otherwise will not open).
The admin can set the visible field of an animal in order to show it on the website or not.

4.7 Account Validation


This functionality is used in the process of registration and autentification.

Checks if the password and username of the current user are found in the data base. This functionality will not let to be more users with the same username or email.

4.8 Create appointments


Real time checking

Technologies used : Ajax.
The user must complete the fields with at least the curent date and curent hour. The system does not allow to make an appointement if the date is invalid.
The user presses the submit button, but has not filled in all the required fields - The system does not allow to make an appointement and warns the user that he has omitted a field that must be filled in.


Functional Requirments

The Ajax call is send to process_visit and datas are sent to the data base in a Json format.
Request from frontend to backend.

4.9 Searching for an animal by filters


Selects multiple animals by checking if the fields selected by the user satify the specific fields of the JSON file corresponding to an animal

4.10 User Session


This functionality check if the user is still logged or not.
If the user is not logged in, he can not use the web site.
